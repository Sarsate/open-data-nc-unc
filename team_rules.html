Code Practices for code review / checkin/out

The team will use the following Gerrit workflow:

First, clone or pull the latest version of the gerrit repository.
git clone http://173.230.133.53:8444/open-data-nc-unc

Checkout your own branch. All changes will be submitted on separate branches and merged into the master branch. DO NOT submit a patch on the master branch; your code will be rejected.
git checkout -b my-branch

Make your changes to the code and commit as usual
git add myfile ( git add --all for all untracked files)
git commit -a

**Make sure your commit message is descriptive of the change. The commit message should encompass the entirety of the patch. If you find yourself making an overly convoluted commit message, you probably need to split your commit into multiple commits. As a guideline, a single sentence is usually not enough. (i.e. "added new page to site" says nothing) **

At this point, you can submit your branch for review on gerrit. If this is your first submission, you'll probably need to set up your credentials:
git review -s

Then simply submit your patch!
git review

If you need to make changes to your patch based on comments/idea from others instead of creating a new commit (or you simply forgot that one semicolon), use git commit --amend:
git commit -a --amend
git review

As long as the Change-ID is the same in the commit message, your new patch will be submitted as a separate patch-set under the same review page as the previous patch, allowing others to easily see and be notified of your changes and you won't have to manually abandon your old change.

Once the patch is submitted the review protocol will take place as follows:
If you are the owner of the patch, you may not review your own patch. You may respond to comments, defend, abandon, or modify your patch, but you cannot +1 or +2 your own commit.

If you are the first reviewer, then you may review the patch with a +1, but cannot +2 the patch until a second reviewer has also +1 or +2'd the same patch.

If a patch already has a +1, and there are no -1 or -2s, then you may +2 the patch and submit the patch. At this point the patch's branch will be merged to the master branch.

Anyone, at anytime (including the submitter), can -1, -2, comment, and make suggestions. In general, a -1 implies that the patch's core is good, but needs a few tweaks before it should be merged (styling, bugs, typos, better implementation etc). A -2 implies that the patch is fundamentally broken/flawed, contrary to our goal, or needs to be seriously reworked. A -2 suggests that the patch should not be merged and likely abandoned.
